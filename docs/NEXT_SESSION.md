## What we did this session (in plain terms)

We fixed the long-running annoyance where every time we built or zipped the extension, the build artifacts would land on disk in ~5 seconds but the build *process* would refuse to die — staying alive forever until we manually killed it with `pkill -f "wxt build"`. Every deploy session for the past two weeks paid this kill-process tax at the fresh-zip step. Today we tracked down the root cause and shipped a clean fix.

Plain-English root cause: the extension build tool (wxt) sits on top of a bundler called Vite, which in version 8 switched from the old JavaScript bundler (Rollup) to a brand-new Rust-based bundler called Rolldown. Rolldown uses native worker threads under the hood for speed — but those worker threads weren't being released when the build finished. Node.js refuses to exit cleanly as long as anything is still holding onto active handles, and the orphaned worker threads were holding the door open. So the build would finish, the dist artifacts would land on disk, the success log would print — and then Node would just sit there forever waiting for something to release the door it was holding open from inside.

Plain-English fix: instead of calling `wxt build` as a separate command (where wxt waits for the door to release before exiting), we now run a small wrapper script that loads wxt's build function directly as a library, calls it, waits for it to finish, and then forcefully tells Node to exit (`process.exit(0)`). Force-exiting is safe because by the time we exit, all the actual work is done — the files are on disk, the log is printed. The only thing happening was a doomed wait for cleanup that never came.

The fix lives in two tiny new scripts (16 lines each) at `extensions/competition-scraping/scripts/wxt-build.mjs` + `scripts/wxt-zip.mjs`, plus four one-line changes to `package.json` to route the `build` + `build:firefox` + `zip` + `zip:firefox` commands through the wrappers. Two `.claude/commands/` slash-command templates (scoreboard.md + deploy.md) had multi-paragraph "KNOWN ISSUE" workaround blocks that documented the old pkill workaround — those got replaced with one-paragraph notes that the wrapper now handles this, with a pointer at `scripts/wxt-build.mjs` if a hang ever regresses.

One scope-expansion judgment call worth flagging: the ROADMAP entry for P-44 mentioned only the `wxt build` hang, but I extended the fix to also cover `wxt zip` because both share the same internal codepath and exhibit the same hang. Fixing build but not zip would have left the deploy session's fresh-zip step still hanging. Symmetric fix across both wrappers is the most-thorough choice per the Default-to-recommendation pattern.
